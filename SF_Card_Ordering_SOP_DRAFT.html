Laundry Card Ordering in Salesforce — Dream-State Requirements

Status: DRAFT for group review  |  Owners: Matt, Trish  |  Requested by: Taylor Panagakis  |  Target: initial draft to hand to the SF team

Purpose: describe the “dream state” for how laundry-card orders should work in Salesforce — what a card order is, and every action that must happen off that order to support fulfillment (the MIND / LHC hand-off). This is a requirements brief, not a build spec; the SF team should treat each section as a capability to design against. Screenshots to be added in a later pass.

1. Background & Problem Statement

Today, laundry-card orders live entirely in MOOPS (our legacy fulfillment system). There is no representation of a card order in Salesforce. The card workflow in MOOPS is mature and well-understood, but it is manual and tightly coupled to MOOPS’ data model. As we move the order system of record toward Salesforce, SF needs to be able to (a) capture a card order, (b) create the card “part” / product that the order references, and (c) trigger the right downstream actions — design approval, purchase orders, and the hand-off to manufacturing/fulfillment — without a human re-keying everything.

The questions the SF team asked us to answer: Is a card a product type that gets ordered? Is it a product with cases/variants? What actions need to fire off the order to support fulfillment? This document answers those at the requirements level.

2. What a “Card” Is (Product Model)

A laundry card is a physical, customer-branded loyalty/payment card produced by a card supplier. In the dream state, SF should model it on two levels:

  • Card as a Product / Order Type. “Laundry Card” should be a first-class product type that can be ordered on its own (a cards-only order) or as a line on a larger equipment/system order. SF needs an explicit order classification for this — equivalent to MOOPS’ “Cards Only” order type — because the downstream actions differ from hardware orders.
  • Card Design as a distinct record (the “part”). Each customer’s card has a unique design that becomes a reusable SKU/part. In MOOPS this is a part record named CARD-MD-{SHORTNAME} cloned from a template. SF needs an equivalent: a Card Design / custom Product record that is created per customer, carries the artwork and approval state, and can be reordered later without re-creating it.

Is it a product with cases/variants? Functionally yes. The same card design is ordered in different quantities and at different price points, and there are a few standing variants the SF team should plan for:

VariantWhat it isImplication for SF
Custom design (per customer)Unique branded card; a new SKU/part is created for the customerTriggers part creation + design approval workflow
Generic cardStock, non-branded card (e.g. CARD-MD-GEN01)No design workflow; small quantities ride along with the equipment order
Reprint / reorderRe-order of an already-approved existing designNo new design; goes straight to a purchase order
Design modifyExisting design with a change (e.g. address update)New proof + approval, but order metadata unchanged

3. Card Order Types & the Action That Each Triggers

The single most important thing for the SF team to internalize: the actions that fire off a card order depend on the design state. There are four paths, and SF needs to route each one differently.

#PathTrigger conditionRequired downstream actions
ANew designCustomer wants a new branded card; no design exists yetCreate the card part/SKU → generate a design proof → route to customer for approval → on approval, create purchase order to the card supplier → ship
BReprint / reorderExisting approved design, customer wants moreNo part creation, no approval. Create purchase order against the existing design → ship
CModifyExisting design with a change (address, logo, etc.)Same as new-design approval flow (new proof → approval → PO), but the order’s classification/tag does NOT change
DGenericStock cards, typically < 1,000 unitsNo design work. Small quantities are bundled onto the related equipment order; no standalone fulfillment

Requirement: SF should capture the design state as a field on the order (or derive it) and drive the right path automatically — ideally with substring-tolerant matching, because the source values are verbose in practice (“Re-print of existing design,” “new design / modify,” etc.).

4. The Card Order Lifecycle (Dream State)

End to end, here is the lifecycle SF should support. Each numbered step is a capability the SF team needs to build or enable.

4.1 Intake & Classification

An incoming request is classified as a card order and routed to the correct path (A–D above). SF should pull: the customer/account, the contact, the card design type, and the card quantity. Where the customer name is missing on the request, it should fall back to the first line of the card description (e.g. “Laundry Depot II card” → “Laundry Depot II”).

4.2 Order Naming / Tagging Convention

Card orders follow a strict naming convention so fulfillment can read them at a glance. SF should generate the order name/tag automatically:

  • Custom: <QTY> Cards (<Customer>) — e.g. “5000 Cards (Pure Wash Laundromats)”
  • Generic: <QTY> Generic Cards (<Customer>)

4.3 Order Type & Shipment Method

For a standalone card order, SF should set the equivalent of: Order Type → Cards Only, and Shipment Method → Drop shipment / Card Supplier (cards ship directly from the supplier, not our warehouse). These should default automatically off the card classification.

4.4 Card Part / SKU Creation (the “card part creation” piece)

This is the step the SF team specifically called out. For a new design, the system creates a new card SKU. In MOOPS this is a clone of a template part with these attributes — SF needs the equivalent fields on its Card Design / Product record:

AttributeDream-state value / behavior
Part / SKU nameAuto-generated: CARD-MD-{SHORTNAME}. Shortname derived from the customer business name — strip vowels, target ~6 chars, fall back to 2-chars-per-word, then acronym, then truncate. (SF can choose its own convention, but it must be deterministic and human-overridable.)
Artwork / imageStarts as a placeholder until the real proof is approved; placeholder state must be queryable (it signals “not yet approved”)
Unit costDefault card cost (currently $0.175 each) — should be a configurable default, not hard-coded
Product group / typeGroup = “Card,” Type = “Virtual” (no physical inventory held by us; supplier produces on demand)
Pricing bookCard pricing list (MOOPS uses USER-CARDS)
OverrideA human can rename the SKU during creation; all later steps must use the actual saved name, not the generated guess

The new SKU is then added to the order matching the source quantity and price. For reprints/modifies, no new SKU is created — the existing one is referenced.

4.5 Design Proof & Customer Approval

For new designs and modifies, a proof must be produced and sent to the customer for sign-off before anything is purchased. Requirements:

  • Generate / attach a design proof and route it to the design team (today this is an email to graphics@mitechisys.com) and to the customer contact.
  • Track an explicit approval state on the order/design. “Placeholder artwork present” = not approved. Approval is the gate for the purchase order — no PO can be created before approval.
  • Notifications on state changes (proof sent, approved) so the team isn’t polling.

4.6 Purchase Order to Card Supplier

Once a design is approved (new/modify) or immediately (reprint), a purchase order is created against the card supplier. Important governance note: in MOOPS today, PO creation is never auto-submitted — a human always reviews and sends. The SF dream state should preserve a human approval gate on the PO even if the rest is automated. The PO flow includes a PO confirmation email to the supplier (with CC handling).

4.7 Quantity-Based Rules

  • Generic < 1,000: no standalone order — bundle onto the related equipment/system order.
  • Orders > 5,000: add a separate shipping line to the order (large runs incur freight that must be billed).

4.8 Hand-off to Fulfillment (MIND / LHC)

This is the open piece and the main reason the SF team needs detail from us. Once the order is classified, the SKU exists, the design is approved, and the PO is cut, SF must hand the order to the manufacturing/fulfillment process (MIND / LHC). The dream state: a structured, machine-readable hand-off that carries everything fulfillment needs — customer/account, ship-to address + ATTN/phone, design SKU + approved artwork reference, quantity, and the PO. We should align as a group on exactly what payload MIND/LHC expects so SF can produce it directly rather than us re-keying into MOOPS. (Flagged as an open question in §6.)

5. Required Data — Fields SF Should Carry

At minimum, a card order in SF should capture and expose:

  • Account / end customer (with the missing-name fallback to card description)
  • Contact (name, email, phone) for design approval
  • Ship-to: company, address, city, state, zip, ATTN name, phone
  • Design state (new / reprint / modify / generic) and approval state
  • Card design SKU + artwork reference (placeholder vs approved)
  • Quantity and unit price
  • Order type (Cards Only vs line on a system order) and shipment method
  • Linked PO + PO state (draft / approved / ordered)
  • Linkage to the parent equipment/system order when the cards ride along

6. Open Questions for the Group

  1. MIND / LHC hand-off payload: what exact fields/format does the manufacturing side expect? This determines the integration contract.
  2. Card design as Product vs custom object: should each customer’s design be a real SF Product (PriceBookEntry) or a custom Card Design object linked to a generic “Laundry Card” product? Affects reporting and reorder UX.
  3. Approval gating: confirm we keep a human gate on PO submission (recommended — guards real spend).
  4. Supplier / drop-ship: single card supplier or multiple? Drives PO routing.
  5. Source of truth during transition: while both systems exist, which is authoritative for card orders, and how do we avoid double-fulfillment?
